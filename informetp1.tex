\section{Diseño e Implementación del programa}


\subsection{Universo}

Al principio habiamos creado la función vecinos para que tome el universo restringido, luego de hablar con el profesor decidimos tomar la teoría del universo infinito (toroidal) que es la que quedo finalmente en el programa.

\subsection{Programa Principal}

El programa principal tiene llamadas a varios métodos responsables de abrir, guardar, leer y cerrar archivos, buscar una posicición dentro de la matriz, imprimir mensajes, etc. La mayor parte del código se realizo en una función  (main) y en un solo archivo (juegodelavida.c) dado que el objetivo principal era conectarse con la función \textbf{\textit{vecinos}}


\subsection{Errores}

El programa valida tanto los parámetros enviados como el archivo de entrada, si hay un error imprime el mismo en el stderr y aborta la ejecución del programa retornando $1$.

Además si el problema es de la cantidad de parámetros el programa muestra la ayuda de como utilizarlo, así como llamar al programa como la opción -h


\subsection{Función Vecinos}

Se decidio que para esta función solo se utilizaría una variable ``v'' (\emph{unsigned int}) que iría acumulando la cantidad de vecinos de la casilla enviada para despues devolverla. Las posiciones serán calculadas a partir de los parametros enviados.

Para obtener una ubicación de la matriz se utilizo la siguiente formula

\begin{center}
$ a[i][j] = a[i*N+j] $
\end{center}

\subsection{Compilación}

Para realizar la compilación de manera mas sencilla usamos un archivo de Makefile\cite{MAKEFILE} para usar junto a la herramienta \textit{make}\cite{MAKE} o \textit{gmake}\cite{GMAKE}

\begin{lstlisting}[language=bash,title={Compilación para usar la función en C}]

$ gmake -f Makefile

\end{lstlisting}

\begin{lstlisting}[language=bash,title={Compilación para usar la función en MIPS32}]

$ gmake -f Makefile asm

\end{lstlisting}

Ambos dejan un programa ejecutable llamado \textbf{conway} que se puede invocar como dice el enunciado.

\subsection{Archivos de Salida}

Se decidio utilizar el formato \emph{Plain PBM}\cite{PBM} para los archivos de salida, aunque de que ocupen mas espacio, dado que es mas sencillo de grabar y revisar su contenido. Este formato utiliza un byte por pixel donde se envia el byte $1$ si el pixel es negro o un byte en $0$ si el pixel es blanco.

Se eligió una celula viva de color negro y una celula muerta de color blanco y no como especificaba el enunciado para que sea más llamativo a la vista el dibujo y no contraste con el fondo.

Se decidió guardar cada celula del juego de la vida como un cuadrado de 16x16 pixeles, como sugería el enunciado, así es mas sencillo verlo en el archivo generado.

\section{Conclusiones}

Una de las cosas que más nos complico fue que en la función en MIPS32 no sabiamos que habia que agregar el .end y el .global para que pueda compilar, eso nos fue ayudando internet y el compilador a la hora de encontrar los errores.

Otra cosa interesante que en C llamar una posición de una matriz o vector nos resulta sencillo pero en MIPS32 hay que hacer mas de un paso para ello.

Menos dificil pero a la vez muy importante al programar en MIPS32 fue que tuvimos que definir muchos rotulos para los saltos, cosa que en C hacemos los ifs/else de manera muy simple y teniamos que ponernos a pensar los nombres de los saltos que no hayamos utilizado ya y que los mismos sean descriptivos de donde saltabamos para saber que iba a hacer y que no nos salteemos nada.
